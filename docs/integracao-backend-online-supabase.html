Apresentação técnica

Como a integração Deefy funciona.

Este material explica a arquitetura que deixa o backend online, conectado ao banco Supabase, protegendo endpoints com JWT e expondo dados para o frontend consumir sem acessar o banco diretamente.

Frontend React → API Spring Boot → PostgreSQL Supabase → Storage público

Regra principal: o frontend conversa com a API Deefy. O banco e as chaves sensíveis ficam no backend/deploy.

Parte 1

O que significa deixar o backend online?

O backend deixa de rodar apenas em localhost:8080 e passa a responder em um domínio público. No nosso caso, o domínio principal serve o frontend e roteia chamadas /api/v1 para o Spring Boot.

Frontend público O navegador abre https://deefy.olua.me e recebe os arquivos estáticos gerados pelo Vite.
API pública O Spring Boot responde em https://deefy.olua.me/api/v1.
Banco privado A conexão PostgreSQL acontece do backend para o Supabase, nunca direto do navegador. 
GET https://deefy.olua.me/api/v1/musics
Authorization: Bearer <JWT_DO_LOGIN>

Parte 2

Como o backend consome o Supabase.

Para o Spring Boot, o Supabase funciona como um PostgreSQL remoto. A aplicação usa JDBC/JPA normalmente, mas as credenciais ficam em variáveis de ambiente no servidor.

Correto Configurar SPRING_DATASOURCE_URL, usuário e senha no .env do deploy. O código Java continua usando repository/service/controller.
Evitar Não colocar senha do banco, service role ou tokens Supabase dentro de arquivos versionados. 
# Exemplo sem segredo real
SPRING_DATASOURCE_URL=jdbc:postgresql://host.supabase.com:6543/postgres?sslmode=require
SPRING_DATASOURCE_USERNAME=postgres.projeto
SPRING_DATASOURCE_PASSWORD=********

SUPABASE_STORAGE_PUBLIC_BASE_URL=https://projeto.supabase.co/storage/v1/object/public
SUPABASE_STORAGE_MUSIC_BUCKET=musicas
SUPABASE_STORAGE_IMAGE_BUCKET=imagens

Parte 3

Login, token e rotas protegidas.

O login retorna um JWT. Esse token precisa ser enviado em toda requisição protegida. Sem isso, o backend responde 401 ou 403, mesmo que a música exista no banco.

Login POST /api/v1/auth/login retorna { token: "..." }.
Salvar token O frontend guarda o JWT localmente para a sessão do usuário.
Enviar token O Axios adiciona Authorization: Bearer token. 
// Exemplo de interceptor no frontend
api.interceptors.request.use((config) => {
  const token = getToken();

  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }

  return config;
});

Parte 4

O frontend não deve saber do banco.

O React precisa conhecer somente a URL da API. Essa URL é definida no build do Vite por VITE_API_URL. Se ela não entrar no build Docker, o site renderiza uma tela preta porque a configuração da API quebra antes de montar a aplicação.

Responsabilidade Onde fica Exemplo
URL pública da API Build do frontend VITE_API_URL=https://deefy.olua.me
Senha do banco Somente backend/deploy SPRING_DATASOURCE_PASSWORD
Service role/storage privado Somente backend/deploy SUPABASE_SERVICE_ROLE_KEY 
# Dockerfile do frontend precisa receber a URL no build
ARG VITE_API_URL
ENV VITE_API_URL=$VITE_API_URL
RUN npm run build

Parte 5

Por que a home ficou sem músicas?

O banco tinha músicas. O problema era contrato de integração: o frontend chamava um endpoint que ainda não existia e também não enviava JWT nas chamadas protegidas.

Antes GET /musics/random?size=12 não existia no backend oficial.
Antes O token era recebido no login, mas não era salvo nem enviado no Axios.
Depois A home chama GET /musics?size=12 e o Axios envia Bearer token. 
// Endpoint existente na API oficial
const response = await api.get('/musics', {
  params: { size: 12 }
});

const musics = response.data?.content || [];

Parte 6

Fluxo de deploy seguro.

O deploy precisa preservar o .env do servidor, reconstruir o frontend com a URL correta e subir backend/frontend apontando para o mesmo contrato.

git push staging → build backend → build frontend com VITE_API_URL → docker compose up 
# Exemplo conceitual
docker compose -f docker-compose.prod.yml up -d --build backend frontend

# Verificações
curl -I https://deefy.olua.me/
curl -I https://deefy.olua.me/swagger-ui/index.html

Checklist para o time

Antes de dizer que quebrou, testar nesta ordem.

Etapa Teste Resultado esperado
Frontend carregou? Abrir https://deefy.olua.me A tela inicial aparece, sem erro de VITE_API_URL.
Login funciona? POST /api/v1/auth/login Retorna token JWT.
Token está sendo enviado? Ver Network no navegador Requisições protegidas têm Authorization: Bearer.
Catálogo funciona? GET /api/v1/musics?size=12 Retorna página com content contendo músicas.
Player toca? Clicar em uma música O áudio usa fileUrl público do Supabase Storage.

Alterações necessárias feitas nesta rodada: salvar JWT no login, enviar JWT no Axios e alinhar a home ao endpoint real de músicas. Não houve alteração de layout ou regra visual do frontend.
