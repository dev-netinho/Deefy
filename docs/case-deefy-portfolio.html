Aprendizado de organizacao

A separacao de grupos funcionou. Ate funcionar demais.

No inicio, dividir responsabilidades parecia a forma certa de ganhar foco. O problema e que a separacao ficou forte o bastante para tambem separar a comunicacao.

Fase A

Quando a divisao virou isolamento

A principio, separar grupos foi eficaz para organizar tarefas. Mas foi tao eficaz que separou demais a comunicacao, inclusive dentro das proprias areas. Isso quebrou a dinamica da Fase A e quase levou a integracao para uma ruina: cada parte evoluia, mas nem sempre com os mesmos contratos, prioridades e entendimento do produto.

Virada

Juncao de Servicos/Negocios com API

Quando pelo menos o grupo de Servicos e Negocios foi unido ao grupo de API, a dinamica melhorou. A comunicacao ficou mais direta, as decisoes ficaram menos espalhadas e a arquitetura em codigo passou a fazer mais sentido como um backend unico.

Efeito pratico

Menos atrito com banco e frontend

Com um backend mais unificado, ficou mais facil conversar com Banco de Dados & Persistencia sobre tabelas, campos e constraints, e tambem ficou mais claro pedir telas e ajustes para o grupo de Frontend.

Como a integracao funciona

O React nao fala com o banco. Ele fala com a API.

Essa foi uma das principais viradas do projeto: parar de tratar telas como prototipos isolados e conectar tudo a um backend que valida regras, autentica usuarios e persiste dados.

React + Nginx Renderiza as telas e usa VITE_API_URL.
Traefik Roteia frontend e /api/v1 para o container correto.
Spring Boot JWT, regras, services, repositories e DTOs.
Supabase PostgreSQL + Storage para imagens e musicas.
Autenticacao

Token real, nao sessao fake

Login gera JWT. O frontend guarda e envia o token nas requisicoes protegidas. Isso permitiu diferenciar usuario comum de admin.

Normalizacao

Compatibilidade entre DTO e player

Alguns campos chegavam como arquivoUrl, enquanto o player esperava outros nomes. A solucao foi normalizar a resposta sem alterar schema.

Performance

Menos queries repetidas

A Home sofria com consultas N+1. Ajustei repositories com carregamento mais eficiente de musica, artista e favoritos.

Backend e persistencia

Como o Spring Boot conversa com o PostgreSQL/Supabase.

O frontend nunca acessa o banco direto. A comunicacao passa pela API: controller recebe HTTP, service aplica regra de negocio, repository consulta o banco via JPA/Hibernate e o mapper devolve DTOs para o React.

Caminho de uma requisicao real
React Chama /api/v1/musics ou /api/v1/playlists com token JWT.
Controller Recebe parametros, valida entrada e chama o service.
Service Aplica regra: buscar artista, evitar duplicidade, montar resposta.
Repository Usa Spring Data JPA para consultar ou salvar entidades.
PostgreSQL Supabase guarda tabelas como musica, artista, playlist e favorito.

Entidade JPA

Classe Java vira tabela

A entidade informa para o Hibernate qual tabela e quais colunas do banco representam uma musica. 

@Entity
@Table(name = "musica")
public class Music {
  @Column(name = "titulo")
  private String title;

  @ManyToOne(fetch = FetchType.LAZY)
  @JoinColumn(name = "artista_id")
  private Artist artist;
}
Repository

Consulta sem SQL manual na maioria dos casos

O Spring Data gera consultas por nome de metodo e tambem permite JPQL quando a busca precisa navegar por relacionamento. 

public interface MusicRepository
  extends JpaRepository<Music, Long> {

  @EntityGraph(attributePaths = "artist")
  Page<Music> findByTitleContainingIgnoreCase(
    String title, Pageable pageable);
}
Service

Regra antes de salvar

O service busca entidades relacionadas, valida existencia e so entao salva. A transacao protege a operacao. 

@Transactional
public MusicDetailResponseDTO createMusic(MusicRequestDTO request) {
  Artist artist = artistRepository.findById(request.artistId())
    .orElseThrow(() -> new ArtistNotFoundException(request.artistId()));

  Music music = musicMapper.toEntity(request);
  music.setArtist(artist);
  return musicMapper.toDetailDTO(musicRepository.save(music));
}
Protecao do schema

ddl-auto=none

O Hibernate mapeia as tabelas, mas nao cria, altera ou remove colunas em producao. Isso evita que a aplicacao mude o banco sem o time de Banco de Dados aprovar.

Performance

@EntityGraph contra N+1

Ao buscar musicas, tambem carregamos o artista quando ele sera usado na resposta. Isso evita uma query extra para cada musica da Home ou dos favoritos.

DTOs

Resposta limpa para o frontend

A API nao devolve a entidade JPA crua. Ela converte para DTOs, deixando o React receber so os campos necessarios, como titulo, artista, capa e arquivo de audio.

Seguranca entre navegador e API

CORS e o porteiro que decide quem pode chamar a API.

CORS foi um dos pontos importantes da integracao: mesmo com backend online, o navegador so deixa o frontend consumir a API quando a resposta do servidor autoriza aquela origem.

Explicacao simples

CORS nao e login. E permissao de origem.

CORS significa Cross-Origin Resource Sharing. Ele existe porque o navegador bloqueia, por padrao, uma pagina tentando acessar dados de outro dominio, protocolo ou porta. A API precisa dizer explicitamente: "essa origem pode falar comigo".

Exemplo no Deefy Frontend: https://deefy.olua.me
API: https://deefy.olua.me/api/v1
Resultado: mesma origem publica, roteada pelo Traefik.
Exemplo em desenvolvimento Frontend: http://localhost:5173
API: http://localhost:8080/api/v1
Resultado: origens diferentes. O backend precisa liberar o localhost.
Cabeçalho esperado Access-Control-Allow-Origin: http://localhost:5173
Access-Control-Allow-Headers: Authorization, Content-Type
Access-Control-Allow-Methods: GET, POST, PUT, PATCH, DELETE, OPTIONS
1. React faz a chamada O navegador envia uma requisicao para a API usando fetch/axios.
2. Navegador confere a origem Se protocolo, dominio ou porta forem diferentes, entra a regra de CORS.
3. API responde com headers O Spring Boot informa quais origens, metodos e headers aceita.
4. Resposta e liberada ou bloqueada Se faltar permissao, o backend pode ate responder, mas o navegador bloqueia o acesso.
Erro real comum

"No Access-Control-Allow-Origin"

Esse erro aparece quando o backend nao inclui o header permitindo a origem do frontend. A correcao e configurar CORS no Spring Security/WebMvc para liberar a URL certa, sem usar * em fluxo autenticado.

Preflight

Por que aparece uma chamada OPTIONS?

Antes de uma requisicao com Authorization ou metodos como PATCH, o navegador faz uma consulta previa. Se o backend bloquear OPTIONS, a chamada real nem chega a acontecer.

Diferenca importante

CORS nao protege a API sozinho

CORS e uma regra do navegador. Quem protege dados de verdade e autenticacao, autorizacao, JWT, validacao de permissao e regras no backend. Ferramentas como Postman nao dependem de CORS como o browser.

Se a API funciona no Swagger mas falha no React, pode ser CORS? Sim. Swagger/Postman podem acessar direto, mas o navegador aplica a politica de origem.
Por que em producao o Deefy usa /api/v1 no mesmo dominio? Para simplificar integracao: o usuario acessa um dominio unico e o Traefik decide se a chamada vai para frontend ou backend.
Posso liberar qualquer origem com *? Em prototipo simples ate parece pratico, mas com token e cookies e arriscado. O correto e liberar apenas origens conhecidas.
Como eu explicaria isso em uma frase? CORS e a permissao que o backend entrega para o navegador saber se aquele frontend esta autorizado a ler a resposta da API.

Admin e catalogo

Popular musica manualmente nao escalava.

Para demonstrar um catalogo real, criamos um fluxo administrativo que importa playlists, reaproveita dados existentes e evita duplicidade.

Pipeline de importacao
Admin Cola URL da playlist e escolhe genero/artista.
Job backend Processo roda em background e registra logs.
yt-dlp + ffmpeg Baixa audio, converte e trata cookies anti-bot.
Supabase Storage Recebe MP3 e capas quando disponiveis.
PostgreSQL Cria artista, musica, playlist e vinculos.
Seguranca

Cookies como segredo operacional

Cookies do YouTube ficam fora do Git, salvos em secrets/, e podem ser atualizados pelo painel admin sem expor senha.

Dados limpos

Deduplicacao antes de inserir

O script procura artista, musica e playlist por nome normalizado antes de criar registros, reduzindo repeticoes como variacoes de acento e caixa.

UX

Playlists globais tocaveis

As playlists importadas viraram colecoes publicas de descoberta, sem botoes de edicao para usuario comum e com player funcional.

Dificuldades e ajustes

Os problemas reais apareceram na integracao.

A maior parte do trabalho dificil nao foi criar telas isoladas: foi fazer frontend, backend, banco, storage e infraestrutura concordarem entre si.

01

Mocks escondiam problemas de contrato

Algumas telas pareciam prontas, mas nao consumiam API real. Foi preciso adaptar services, autenticar requests e alinhar DTOs.

02

Home lenta por N+1 queries

O gargalo nao era a VPS; era o backend buscando artistas e favoritos em consultas repetidas. Corrigimos sem mexer no banco.

03

YouTube bloqueou a VPS

O importador precisou lidar com HTTP 429, cookies, runtime JavaScript, EJS e logs maiores para diagnosticar falhas.

04

Cache e permissao quebravam a experiencia

Adicionamos no-cache no index.html e protegemos playlists globais para usuario comum nao editar o que e publico.
